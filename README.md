# Taskora Real MVP v3 - Railway + PostgreSQL

هذه نسخة حقيقية أولية من Taskora، وليست مجرد Frontend.

## المزايا

- تسجيل حساب حقيقي:
  - `username` فريد
  - البريد فريد
  - الهاتف فريد
  - كلمة المرور 8 خانات على الأقل وتحتوي حروفًا وأرقامًا
- تشفير كلمات المرور باستخدام `bcrypt`
- JWT Auth
- PostgreSQL Database
- KYC:
  - هوية أو جواز سفر
  - منع تكرار نفس رقم الوثيقة
  - منع تكرار نفس `document_hash`
- بونص واحد فقط لكل هوية بعد قبول التوثيق من الأدمن
- لا يمكن السحب إلا إذا كان الحساب `verified`
- طلبات إيداع بمراجعة الأدمن
- طلبات سحب بمراجعة الأدمن
- حجز مبلغ السحب عند إرسال الطلب
- لوحة أدمن لإدارة:
  - المستخدمين
  - التوثيق
  - الإيداعات
  - السحوبات
- باقات ومهام يومية
- Railway-ready

## التشغيل المحلي

```bash
npm install
cp .env.example .env
npm start
```

يجب وضع `DATABASE_URL` في `.env`.

## الرفع على Railway

1. ارفع محتويات هذا المشروع إلى GitHub.
2. افتح Railway.
3. اختر:
   ```text
   New Project > Deploy from GitHub Repo
   ```
4. أضف PostgreSQL من Railway:
   ```text
   Add Service > Database > PostgreSQL
   ```
5. في خدمة التطبيق أضف المتغيرات:

```env
DATABASE_URL=${{Postgres.DATABASE_URL}}
JWT_SECRET=ضع_سر_طويل_عشوائي
ADMIN_USERNAME=admin
ADMIN_EMAIL=admin@taskora.app
ADMIN_PHONE=0000000000
ADMIN_PASSWORD=Admin12345
NODE_ENV=production
```

Railway PostgreSQL يوفر متغير `DATABASE_URL` للتوصيل من التطبيق.

## تحديث v10.1 - إصلاح Railway Healthcheck

هذه النسخة تصلح مشكلة:

```text
Network > Healthcheck failure
service unavailable
```

التعديل:

- السيرفر يبدأ الاستماع على `PORT` فورًا.
- `/health` يرجع `200 OK` حتى لو قاعدة البيانات غير متصلة.
- حالة قاعدة البيانات تظهر داخل استجابة `/health`.
- فشل migrations لا يقتل التطبيق أثناء Healthcheck، بل يظهر في Runtime Logs.

بعد النشر افتح:

```text
/health
```

إذا ظهر `database: "disconnected"` فالمشكلة في `DATABASE_URL` أو PostgreSQL، لكن النشر لن يفشل بسبب Healthcheck.


## تحديث v10.2 - إصلاح التسجيل وتسجيل الدخول

سبب مشكلة التسجيل كان ترتيب الـ migrations:

- كان يتم إنشاء فهرس على جدول `notifications` قبل إنشاء الجدول نفسه.
- لذلك كانت migrations تفشل قبل إضافة أعمدة مهمة مثل `email_verification_token`.
- النتيجة: `/health` يظهر database connected، لكن `/api/auth/register` يفشل.

تم الإصلاح:

- نقل إنشاء أعمدة المستخدم المهمة مباشرة بعد إنشاء جدول `users`.
- إنشاء جدول `notifications` قبل إنشاء الفهرس الخاص به.
- إضافة حالة migration داخل `/health`:
  - `migration: "completed"`
  - أو `migration: "failed"` مع `migration_error`.

بعد رفع هذه النسخة افتح:

```text
/health
```

يجب أن ترى:

```json
"database": "connected",
"migration": "completed"
```
